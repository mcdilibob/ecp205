# Промпт для Claude Code: рефакторинг GUI-проекта

## Контекст проекта

В текущей директории находятся два проекта:

1. **Мой проект** (корень директории) — GUI-программа под Ubuntu (Linux), запускается через скрипт.
   - 1 микроконтроллер, связь напрямую (UART/Serial)
   - Функционал: окно графиков с установки, окно АЧХ, задание управления, настройка параметров

2. **`Gyro_main/`** — проект коллеги под Windows (.exe). Похожая установка, но:
   - 2 микроконтроллера, связь по CAN
   - Более проработанный UI/UX, графики, архитектура кода

`Gyro_main` — это **референс**, не зависимость. Из него нужно заимствовать подходы, а не код напрямую.

---

## Задача

Провести рефакторинг моего проекта, заимствуя лучшие архитектурные и стилистические решения из `Gyro_main`.

---

## Фаза 0 — Анализ (ОБЯЗАТЕЛЬНО ПЕРЕД ЛЮБЫМИ ИЗМЕНЕНИЯМИ)

Прежде чем писать или менять код:

1. **Изучи оба проекта целиком** — структуру файлов, точки входа, ключевые классы/модули.
2. **Составь и выведи краткий отчёт:**
   - Структура моего проекта: файлы, модули, как запускается, какой GUI-фреймворк
   - Структура `Gyro_main`: файлы, модули, GUI-фреймворк, паттерны
   - Таблица сравнения: что сделано лучше в `Gyro_main`, что можно перенять
   - Что из `Gyro_main` **нельзя** перенести напрямую (CAN, Windows-специфика) и какие альтернативы предлагаются
3. **Предложи план рефакторинга** — список конкретных шагов с приоритетами.
4. **Дождись моего подтверждения** перед началом изменений.

---

## Фаза 1 — Рефакторинг (после подтверждения)

### Что ОБЯЗАТЕЛЬНО сохранить (не сломать):
- Окно с графиками реального времени с установки
- Окно АЧХ (амплитудно-частотная характеристика)
- Возможность задания управления микроконтроллером
- Настройка параметров установки
- Совместимость с Linux (Ubuntu) — проект должен запускаться как раньше
- Текущий интерфейс связи (UART/Serial) — CAN не используется

### Что перенести/адаптировать из `Gyro_main`:
- **Архитектура GUI** — разделение логики и представления (MVC/MVP или как реализовано в референсе)
- **Стиль интерфейса** — расположение элементов, визуальное оформление (адаптировать кроссплатформенно)
- **Построение графиков** — подход к обновлению, масштабированию, отображению данных
- **Хранение и обработка данных** — буферы, структуры данных, конвертация
- **Паттерны взаимодействия GUI ↔ MCU** — адаптировать под 1 контроллер и UART/Serial

### Чего НЕ делать:
- Не менять интерфейс связи на CAN
- Не добавлять логику второго микроконтроллера
- Не использовать Windows-only библиотеки или API
- Не копировать код механически — адаптировать, делая чище
- Не менять точку входа / способ запуска без согласования

## Фаза 2 — Документация

После внесения изменений предоставь:

1. **Обновлённую структуру проекта** (дерево файлов)
2. **Список изменений** — что именно изменено и почему
3. **Маппинг заимствований** — где и какие идеи взяты из `Gyro_main`
4. **Рекомендации** — что ещё можно улучшить в будущем

## Правила работы

- Вноси изменения **поэтапно**, по одному логическому блоку за раз
- После каждого крупного изменения — кратко поясни, что сделано
- Если что-то из `Gyro_main` невозможно перенести — предложи Linux-эквивалент
- Если не уверен в решении — спроси, не угадывай
- Сохраняй рабочее состояние проекта на каждом этапе (проект должен запускаться)
